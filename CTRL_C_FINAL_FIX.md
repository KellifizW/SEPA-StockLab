🔧 最終解決方案 (app.py 3760-3830 行):
─────────────────────────────────────────────────────────────────────────────

而不是直接用 app.run() 阻塞主線程，我們現在：
1. 在 daemon 線程中運行 Flask 服務器
2. 主線程執行 sleep 循環，定期休眠 1 秒
3. 主線程可以正確接收和處理 SIGINT/SIGTERM 信號
4. 當收到信號時調用 _cleanup_and_exit(0) 使用 os._exit()

流程圖:
┌─ Main Thread ─────────────────────┐
│ 接收信號 (SIGINT/SIGTERM)         │
│ ↓                                 │
│ _cleanup_and_exit() 調用          │
│ ↓                                 │
│ os._exit(0) ← 立即終止進程        │
│                                  │
│ ↑ 主線程 sleep(1) 循環            │
└──────────────────────────────────┘

┌─ Flask Thread (daemon) ───────────┐
│ app.run() 処理 HTTP 請求          │
│ 當主線程調用 os._exit() 時自動    │
│ 被終止（daemon thread）           │
└──────────────────────────────────┘

💡 技術細節:
─────────────────────────────────────────────────────────────────────────────

為什麼之前的方案不工作？
• app.run() 直接呼叫 Werkzeug 的 serving loop
• Werkzeug 可能有自己的信號處理器，干擾或攔截信號
• Windows 上 Ctrl+C 被轉換為 CTRL_C_EVENT，進一步增加複雜性
• app.run() 本身會阻塞，無法立即處理信號

為什麼這個方案有效？
• 主線程执行簡單的 sleep 循環
• 睡眠時間短（1 秒），允許快速的信號檢測
• os._exit() 在 Windows 和 Unix 上都支持，強制立即終止
• 不依賴 Flask/Werkzeug 的信號處理

app.py (lines 3760-3830):
